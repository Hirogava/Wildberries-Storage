# Архитектура backend-сервиса

## Роль сервиса

Go-сервис является оркестратором бизнес-процесса:

1. принимает точки прогнозирования;
2. вызывает внешний ML/AI-сервис по HTTP;
3. превращает прогноз в операционное решение по вызову транспорта;
4. формирует submission-файл для соревнования;
5. считает метрики качества по офлайн-оценке.

## Компоненты

- `cmd/server` — точка входа.
- `internal/httpapi` — HTTP-слой и JSON-контракты.
- `internal/service` — бизнес-логика.
- `internal/ml` — клиент для внешнего ML-сервиса.
- `internal/storage` — чтение входных файлов и запись submission CSV.
- `internal/ui` — встроенный web-интерфейс для ручной проверки API.
- `ml_service` — минимальный Python stub для локального стенда.

## Потоки

### Онлайн-контур

`client -> /predict -> ML service -> /decision -> truck request`

### Офлайн-контур

`client -> /batch -> ML service -> submission.csv`

### Контур оценки

`client -> /metrics -> WAPE + Relative Bias`

### Контур демонстрации

`browser -> Go UI -> Go API -> ML stub`

## Бизнес-допущения MVP

- все машины считаются одинаковой вместимости;
- решение принимается на основании прогноза на ближайшие 2 часа;
- для защиты от недогруза применяется safety factor;
- количество машин ограничивается `max_trucks_per_route`;
- отрицательные прогнозы принудительно обрезаются до нуля;
- внешний ML-сервис отвечает за выбор модели, feature engineering и inference.
- `/decision` умеет восстанавливать `route_id` и `timestamp` по `id`, если переданы компактные `predictions` из `/predict` и исходные `points`.
