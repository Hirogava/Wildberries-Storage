О соревновании
Эффективная логистика невозможна без точного планирования отгрузок и своевременного вызова транспорта. Ошибки в этих решениях приводят к простоям складов, задержкам заказов и росту операционных затрат.

В командном треке участникам предстоит спроектировать систему автоматического вызова транспорта на склады на основе прогноза отгрузок.

Задача включает:

Создание предсказательной модели на основе предоставленных тренировочных данных 

Разработка архитектуры и бизнес-логики работы сервиса-системы автоматического вызова транспорта на склады на основе прогноза отгрузок.

Требования к системе автоматического вызова транспорта 

В командном треке участникам необходимо предложить прототип системы автоматического вызова транспорта на склады на основе прогноза отгрузок. Разрабатываемая система должна обеспечить логический переход от получения прогноза объемов отгрузок к принятию конкретных операционных решений - например, к расчету необходимого количества транспорта и формированию соответствующих заявок.  

В основе разрабатываемой системы должна лежать созданная вами прогнозная модель или несколько моделей. Прототип системы в общем виде должен представлять из себя веб-сервис или связку из нескольких веб-сервисов, выполненных на любом языке программирования с использованием любых доступных открытых технологий. В рамках создания прототипа системы вам необходимо: 

Определить логику работы продукта с прогнозом: как вы будете использовать прогнозные данные и с какой периодичностью отдавать прогнозы, какие преобразования данных можете предложить, на какой горизонт планирования вы наиболее уверены в прогнозах и т.д.

Предложить подход к оценке качества системы: какие бизнес-метрики будете использовать для оценки решения, как будете тестировать, с каким бейзлайном сравнивать

Предложить пути развития продукта: как его можно масштабировать и какие дополнительные данные на ваш взгляд можно использовать для улучшения прогноза и качества работы вашего продукта.

Разработать архитектуру и оформить репозиторий github с кодом, в котором ваше решение будет подробно описано с бизнесовой и архитектурной точки зрения. Обязательно включите в репозиторий файл README.md и опишите все шаги по запуску вашего решения, а также принятые вами бизнес-допущения (например, предположение о том, что все машины одинакового объема или любые другие), чтобы мы могли по достоинству оценить его.

Для расчета метрики будет использоваться следующий код:

```

# Текущая реализация метрики на платформе

class WapePlusRbias:

    """Calculates as WAPE + Relative Bias."""

    def calculate(self, y_true: np.ndarray, y_pred: np.ndarray) -> float:

        """Рассчитывает значение метрики."""

        wape = (np.abs(y_pred - y_true)).sum() / y_true.sum()

        rbias = np.abs(y_pred.sum() / y_true.sum() - 1)

        return wape + rbias

```


Для автоматического расчета метрики участникам необходимо загрузить на платформу файл с решением в формате *.csv, в котором для каждой строки из тестовой выборки (файл test.csv) имеется строго две колонки:

id - уникальный идентификатор строки из тестовой выборки

y_pred - предсказанное значение целевой переменной

Файл baseline.ipynb закреплен в разделе “Ноутбуки” и содержит baseline (базовое решение) задачи, которое можно использовать в качестве примера. Скачать можно в разделе “Ноутбуки”. Он также содержит пример формирования правильного файла решения.

Файл sample_submission.csv содержит сформированный пример решения в корректном формате.

Для оценки сервиса необходимо предоставить github репозиторий соответствующий требованиям.


Участникам предлагается тренировочный набор данных в формате .parquet, содержащий следующие колонки: 

route_id - уникальный идентификатор маршрута

office_from_id - склад, из которого идет данный маршрут. Привязка маршрутов к складам в рамках задачи не изменяется

timestamp - момент времени

status_1, status_2, status_3, status_4, status_5, status_6, status_7, status_8 - количество товаров, направляющихся по данному маршруту, прошедших конкретные этапы обработки на текущем складе за последние 30 минут

target_2h - целевая переменная. Кол-во емкостей, отгруженных по маршруту за последние два часа.

Тестовый набор данных в формате .parquet не содержит данных о статусах и целевой переменной. Ваша задача предсказать значение целевой переменной для каждой временной точки из тестового набора.
